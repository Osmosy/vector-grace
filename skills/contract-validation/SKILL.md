# Contract Validation — делегирование с контрактом

## Суть

Обычное делегирование: «сделай X» → агент делает → «готово».
Проблема: агент считает «готово» когда код компилируется, а не когда
задача решена по существу.

Контракт валидации добавляет формальный шаг:
  1. Оркестратор формирует contract (checklist критериев)
  2. Worker выполняет задачу
  3. Валидатор проверяет результат по contract
  4. Если не пройдено — worker дорабатывает

## Когда использовать

- Любой delegate_task где важен результат, а не процесс
- Задачи где «работает» ≠ «готово» (код компилируется но бизнес-логика неверна)
- Многошаговые задачи где ошибка на шаге 2 ломает шаг 5
- Когда цена ошибки высока (production, data, конфигурация)

## Workflow

### Шаг 1 — Формулировка контракта (оркестратор)

Перед делегированием сформулируй contract — 3-7 критериев
которые ОДНОЗНАЧНО проверяемы. Избегай субъективных:
«хороший код» — плохо, «все тесты проходят» — хорошо.

Формат:
```
CONTRACT:
1. [критерий] — как проверить
2. [критерий] — как проверить
...
```

Примеры критериев:
- Все тесты проходят: `pytest -x`
- Файл создан: `stat /path/to/file`
- JSON валиден: `python3 -c "import json; json.load(open('file'))"`
- Ответ содержит конкретные поля: `jq '.answer' response.json`
- Порт отвечает: `curl -sI http://localhost:PORT`

### Шаг 2 — Делегирование (worker)

```
delegate_task(
  goal="[задача + CONTRACT в контексте]",
  context="CONTRACT:\n[критерии]\n\nПосле выполнения — верифицируй себя по контракту перед тем как считать работу готовой.",
  toolsets=["terminal", "file", "web"]
)
```

Worker должен САМ проверить контракт перед возвратом результата.
Это первый уровень валидации.

### Шаг 3 — Валидация (validator)

После получения результата от worker — запусти валидатор
(второй delegate_task или сам оркестратор):

```
delegate_task(
  goal="Проверь результат работы по контракту. Каждый критерий — проверь и отметь PASS/FAIL.",
  context="CONTRACT:\n[критерии]\n\nRESULT:\n[результат worker]",
  toolsets=["terminal", "file"]
)
```

### Шаг 4 — Решение

- Все PASS → задача завершена
- Есть FAIL → верни worker с указанием какие критерии не пройдены
- Если FAIL после 2 итераций → escalate пользователю

## Пример

Задача: «Создай конфиг для нового MCP-сервера»

```
CONTRACT:
1. Файл существует: stat ~/.hermes/config.yaml
2. Новая секция есть: grep "new-server" ~/.hermes/config.yaml
3. YAML валиден: python3 -c "import yaml; yaml.safe_load(open(...))"
4. Сервер отвечает: curl -s http://localhost:PORT/health
5. Hermes видит сервер: hermes mcp list | grep new-server
```

## Анти-паттерны

- «Выглядит хорошо» вместо проверяемого критерия
- Контракт из 15+ пунктов — разбей задачу
- Пропускать валидацию потому что «я уверен»
- Валидатор = тот же worker (должен быть свежий контекст)

## Интеграция с другими skills

- `doubt-driven-review` — используй КАК валидатор для критичных задач
- `verification-before-completion` — дополняет контракт на финальном шаге
- `grm-self-evaluation` — worker может использовать для самооценки
