# GRACE: AI-оператор и смерть традиционных форм

Фундаментальный сдвиг в проектировании фронтенда, вызванный появлением AI-агентов как полноправных участников системы.

## Аксиома

> Наличие AI в системе — Game Changer. AI меняет структуру вводимой информации и сам use case взаимодействия.

Старые фронтенды проектировались под парадигму «человек заполняет форму». AI-оператор ломает эту парадигму: данные вводит не человек, а AI по результатам обсуждения с человеком.

## Как меняется use case

| Было (до AI) | Стало (с AI-оператором) |
|---|---|
| Заполни форму | Обсуди с AI → проверь как AI заполнил данные → утверди |
| Человек — источник данных | AI — источник данных, человек — валидатор |
| Формы для каждого поля | Дашборды и визуализаторы последствий |
| Интерфейс = набор форм | Интерфейс = диалог + визуализация |

## Три правила проектирования AI-фронтенда

### 1. Не проектируй формы для экспертных параметров

Экспертные параметры (например, «Приоритет» в ERP) — это то, что человеку сложно и поставить. Их заполняет ТОЛЬКО AI-агент. Человек только проверяет.

**Следствие:** для таких параметров не нужны формы ввода. Нужны дашборды и визуализаторы, показывающие последствия этих данных.

**Антипаттерн:** «Давайте сделаем как раньше, а потом добавим AI». Старый фронтенд с гиперакцентом на формы рассыпается — AI заполняет их сам, формы становятся мёртвым грузом.

### 2. Строй интерфейс вокруг диалога

AI-оператор общается с человеком на естественном языке. Чтобы ввести данные, человеку нужно... общаться с AI. Чат-интерфейс становится первичным, визуальные элементы — вторичными (подтверждение и проверка).

```
[Чат] ←→ [AI-оператор] → [База данных]
                ↓
         [Дашборд: что изменилось]
                ↓
         [Человек: проверка и утверждение]
```

### 3. Разделяй поля на три категории

При проектировании схемы данных и фронтенда:

| Категория | Кто заполняет | Интерфейс |
|---|---|---|
| **Фактические** (имя, email) | Человек или AI | Лёгкая форма (опционально) |
| **Экспертные** (приоритет, категория, оценка) | ТОЛЬКО AI | Только визуализация, без форм |
| **Утверждаемые** (статус, флаг) | AI предлагает → человек утверждает | Дашборд с кнопкой «Подтвердить» |

## Когда применять

- Проектирование ERP, CRM, систем управления ресурсами с AI-компонентой
- Миграция старой системы на AI-агентную архитектуру
- Оценка: стоит ли сохранять старый фронтенд при добавлении AI
- Проектирование интерфейсов для sakhranova.ru, Vector Work, любых систем с AI-оператором

## Когда НЕ применять

- Системы без AI-компоненты (чисто человеческий ввод)
- Простые лендинги и витрины (там нет оператора)
- Системы, где AI — вспомогательный инструмент, а не оператор данных

## Связь с другими GRACE-модулями

- `grace-plan` — при проектировании модулей учитывать разделение полей на категории
- `grace-init` — при создании нового проекта закладывать AI-фронтенд-архитектуру с первого дня
- `grace-verification` — тестировать не формы, а корректность AI-заполнения и визуализацию

## Пример: ERP для школы фен-шуй (sakhranova.ru)

Традиционный подход:
```
Форма «Новый ученик» → 15 полей → человек заполняет → сохранить
```

AI-подход:
```
Чат: «Добавь ученика Иванову Анну, родилась 15.03.1985»
AI: «Добавил. Определил: элемент — Дерево, благоприятное направление — Восток.
     Рекомендую курс "Основы Бацзы". Подтверждаешь?»
Человек: «Да, но курс поменяй на "Продвинутый фен-шуй"»
AI: «Обновил. Вот карточка ученика: [дашборд с данными]»
```

Ни одной формы. Только диалог и визуализация.
