# Gotcha: Документация в MD-файлах vs встроенная в код

## Эксперимент (Владимир Иванов, GRACE)

На учебном проекте (Python бэкенд + React SPA) сравнивались два подхода к документации:

### Отдельные MD-файлы + agents.md

- Пока приложение простое — документация не нужна, агенты и так справляются
- Когда сложность растёт — агенты ИГНОРИРУЮТ MD-документацию в ~50% случаев
- MCP Context7 игнорируется в 100% случаев
- Агенты охотно используют Tavily (паттерн «погуглить»), но не «пойти почитать документацию»
- Разные сессии создают snapshot-отчёты, которые описывают приложение которого УЖЕ НЕТ
- Накопление таких отчётов создаёт у агента ложные гипотезы о текущем состоянии

### Встроенная документация в коде (GRACE-стиль)

- Всегда актуальна (меняется вместе с кодом)
- 100% гарантия прочтения — агенту деваться некуда
- Doxygen/JavaDoc-паттерн + AI-friendly контракты (MODULE_CONTRACT, START_BLOCK/END_BLOCK)
- Работает как для людей, так и для агентов

## Почему агенты избегают MD-документацию

Гипотеза: на Reinforcement Learning агенты многократно наблюдали «бардак» в документации —
противоречивые, устаревшие, нерелевантные MD-файлы. Модель learned behaviour «если проблема
выглядит простой — не читай документацию, она скорее всего бесполезна».

Tavily работает потому что паттерн «поискать в интернете» встроен в training distribution
как часть стандартной траектории решения проблемы. «Прочитать локальную документацию» —
нет, это не часть стандартной траектории.

## Следствия для GRACE

1. **Документация живёт в коде.** Не в отдельных MD-файлах. Не в snapshot-отчётах сессий.
2. **OKF-паттерн требует дисциплины.** Если используешь structured Markdown — он должен
   обновляться СИНХРОННО с кодом, а не быть артефактом сессии.
3. **session_search — для recall, не для документации.** История сессий полезна для
   восстановления контекста, но не как source of truth.
4. **AGENTS.md должен ссылаться на ЖИВЫЕ файлы**, а не на историю изменений.
