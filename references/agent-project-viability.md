# Agent Project Viability: Intelligence vs Judgment

Source: Всеволод Викулин, «Реальная автоматизация» (vikulin.ai)

## Когда GRACE применим, а когда — нет

GRACE работает для **intelligence-задач** — где есть протокол, регламент, критерии хорошего результата. Если регламент не написать — это judgment-задача, агент не поможет в принципе.

Простой тест: сядь и напиши регламент — что делать, куда ходить, как принимать решения, что считать хорошим результатом. Если получилось — intelligence, GRACE применим. Если не получилось — либо в процессе бардак (фикси бардак, не агента), либо это judgment (GRACE неприменим).

## Три стадии разработки

1. **Документация + метрики** — до первой строчки кода: выписать регламенты процесса, подготовить инструменты с описаниями, рассчитать KPI-метрики. Без метрик — нет способа оценить, работает ли агент.

2. **Прототип на лучшей модели** — взять самую дорогую/сильную API-модель, собрать архитектуру, проверить гипотезу. Здесь запрещено думать про стоимость инференса и оптимизации. Каждое изменение проверяется метриками. Если прототип не завёлся — проект честно закрывается.

3. **Оптимизация + дистилляция** — сигнал получен, гипотеза подтвердилась. Теперь оптимизируем инференс, применяем дистилляцию (маленькая модель повторяет ответы большой на конкретных задачах). Сравниваем с прототипом по метрикам.

## Четыре уровня контроля (пропорционально автономности и риску)

1. **Observability** — трейсы, логирование тулов, метрики качества, дашборды. Без этого разбор ошибок занимает вечность.
2. **Guardrails** — LLM-критик в рантайме, оценивающий действия агента по тем же критериям, что eval-метрики. Блокирует плохие ответы до доставки пользователю.
3. **Security barriers** — минимум инструментов, контроль доступов (агент видит только данные своего клиента), маскирование PII, защита от prompt injection.
4. **Human-in-the-loop** — эскалация на человека по триггерам: критическое решение по регламенту, guardrail сомневается, агент сам просит помощи.

## Связь с GRACE

- Intelligence-задача = контракт можно написать → GRACE применим
- Judgment-задача = контракт не пишется → GRACE неприменим, проект обречён
- GRACE Verification Plan = уровень 1 (Observability) + уровень 2 (Guardrails)
- GRACE Execution Profiles (safe/balanced/fast) = соотносятся с уровнями автономности и контроля
- PCAM Autonomy Gate = уровень 4 (Human-in-the-loop) для safe-профиля

## Критерии отсечения проектов

1. **Профит >> инфраструктура контроля** — если эффект от автоматизации не превышает стоимость контроля в 3-5 раз, проект не окупится
2. **Интеллект, не экспертность** — регламент пишется → intelligence → агент справится. Не пишется → judgment → агент не поможет
3. **Ресурсы выдержат риск** — для процессов под надзором регулятора стоимость контроля может превысить профит. Отсекаем раньше
