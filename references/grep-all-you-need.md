# «Is Grep All You Need?» — PwC, май 2026

Sen et al., PricewaterhouseCoopers. arXiv:2605.15184.

## Ключевой результат

**Векторный поиск всегда проигрывает grep на любом агенте и на любой LLM** в стандартном сценарии вызова поиска с получением результата напрямую.

Векторный поиск может быть эффективнее **только** когда агент сначала сохранит результат в JSON/XML-файл, а потом будет делать в нём grep.

## Экспериментальный дизайн

- **116 вопросов** из LongMemEval
- **4 harness'а:** Chronos (свой), Claude Code, Codex, Gemini CLI
- **2 режима:** inline (результат прямо в контексте) vs file-based (результат в файл → агент читает)
- **2 эксперимента:**
  1. grep vs vector на всех harness'ах и режимах
  2. grep vs vector при увеличении шума (посторонняя история диалога)

## Почему grep побеждает

Агентам нужна **не приблизительная точность, а полные совпадения** — особенно при работе с кодом. Векторный поиск даёт «похожие» результаты, но агент ищет конкретный идентификатор, функцию, строку.

## Связь с GRACE

Автор GRACE сделал ставку на grep **до этой статьи**, на основе разбора типичных траекторий агентов:

- LLM делают **максимальную ставку** на полнотекстовый поиск
- Другие инструменты работают хуже или **отвергаются агентом**, если есть выбор применить grep
- В GRACE 2.0 grep-стратегия первична для навигации по коду

**Основной вывод статьи**: агентам кодирования **кроме grep в обычных кейсах навигации по коду практически ничего не нужно**.

## Практические следствия

1. Инструмент `search_files` (ripgrep-based) > векторный RAG для агента
2. Если векторный поиск неизбежен — сохранять результат в файл и делать grep по файлу
3. GRACE-разметка (`START_BLOCK`, `END_BLOCK`, уникальные XML-теги) создана под grep: каждый якорь — уникальная строка для точного поиска
4. Отказ от agents.md и переход к in-code markup — это не только снижение перплексии, но и создание grep-дружественной структуры

## Комментарий автора GRACE: аудиторское заключение, не дискуссия

> «Вы пишите слова, а PwC доказательства. Вы ошиблись сделав ставку на векторный поиск — это доказано научным способом на всех LLM и разных агентах.»

### Почему это тотальная архитектурная ошибка

- Проигрыш векторного поиска воспроизводится на **всех LLM и всех агентах** — это исключает «особенности реализации»
- Тотальный характер ошибки означает, что проблема не в конкретном вендоре, а в самой идее векторного поиска для кодовой навигации

### Коренная проблема векторного поиска

**Нет языка запросов.** Grep позволяет OR, AND, regex, инвертированные паттерны. Векторный поиск вынуждает агента начать перечитывать файлы, чтобы понять, то ли он нашёл.

Ставка на векторный поиск делается не из инженерных соображений, а из моды: «вектор — это круто». Это глупое обоснование для продакшен-системы.

## Комментарий автора GRACE: код — не бизнес-помойка

> «Векторный поиск больше "хайповая тема" и продается через "вектор — круто", а что круто никто не доказал, а если доказывать, то видим обратное.»

### Ошибочный перенос RAG с документов на код

**Основная ошибочная гипотеза**: если векторный RAG работает на бизнес-помойке документов, то будет работать и на коде.

Это логическая ошибка. Природа информации принципиально разная:

| | Бизнес-документы | Код |
|---|---|---|
| Природа | Размытая, синонимы, paraphrasing | Жёсткие идентификаторы |
| Поиск | «Похожие» документы ок | Нужно точное совпадение |
| Язык запросов | Не критичен | Критичен: OR, AND, regex |
| Инструмент | Векторный RAG | grep/ripgrep |

Код — **строгая информация**. Весь код строится на жёстких идентификаторах: имена функций, классы, переменные, импорты, строковые литералы, ключи конфигов. За них grep и цепляется.

### Автор GRACE — эксперт по векторизации

GRACE использует продвинутое обогащение чанков семантикой. Но **по факту агент предпочитает за эти keywords цепляться через grep**, а не через векторный поиск по embedding'ам этих же чанков. Обогащение семантикой полезно — но как источник grep-ключей, не как векторный индекс.

### Масштаб ошибки

Claude Code (~5 млн пользователей) отказался от векторного поиска. Мнение «это 0,1% базы» — математически неверно и статистически незначимо на таком масштабе. Anthropic принял архитектурное решение на основе данных, и PwC подтвердил его правильность независимым исследованием.
