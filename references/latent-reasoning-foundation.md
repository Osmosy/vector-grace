# Теоретическое основание GRACE: латентное рассуждение LLM

**Источник:** Wenshuo Wang, "LLM Reasoning Is Latent, Not the Chain of Thought"
**arXiv:** https://arxiv.org/abs/2604.15726
**Дата:** 17 Apr 2026

## Три гипотезы

Автор разделяет три склеенных фактора, которые в 99% работ по reasoning смешаны:

| Фактор | Обозначение | Что это |
|---|---|---|
| Surface CoT | **SS** | Явный текст рассуждений, который мы видим |
| Latent trajectory | **ZZ** | Внутренняя траектория скрытых состояний, несущая промежуточные commitments |
| Serial compute | **BB** | Дополнительный вычислительный бюджет (больше токенов = больше шагов) |

### H1 — Latent-trajectory mediation (подтверждена)
Рассуждение первично опосредовано латентными траекториями ZZ. Surface CoT — лишь частичный интерфейс. Может иногда сообщать о латентных commitments, может их возмущать, но не является объектом рассуждения как таковым.

### H2 — Surface CoT mediation (опровергнута как общий случай)
Рассуждение первично опосредовано явным текстовым CoT. ZZ — лишь эхо SS.

### H0 — Generic serial compute (работает только при гигантском бюджете)
Большинство улучшений рассуждения лучше объясняются увеличением последовательного compute (BB), чем какой-либо привилегированной репрезентацией (ни ZZ, ни SS).

## Ключевая проблема

CoT-промптинг меняет **одновременно** SS и BB. Latent reasoning методы — ZZ и BB. Test-time scaling — BB и output path. Результаты невозможно интерпретировать как подтверждение одной гипотезы.

## Результаты экспериментов

- **H1 побеждает** в ordinary regimes (обычный режим работы)
- **H2 всплывает** как boundary case, когда visible trace конститутивен для задачи (пошаговая верификация в математике)
- **H0 всплывает** как boundary case, когда serial compute доминирует (гигантский budget на search/sampling)

**Итог:** H1 — рабочая гипотеза по умолчанию, но не task-independent verdict.

## Маппинг на GRACE

### 1. Belief State — не опция, а необходимость

Если ZZ невербализуем, structured logging с семантическими якорями — единственный способ сделать рассуждение агента наблюдаемым.

```
[DEBUG][load_and_parse_data][PARSE_DATETIME_COLUMNS][Attempt] Парсинг столбцов дат
```

Без этого мы видим post-hoc рационализацию (CoT), а не реальный процесс (ZZ). GRACE-логирование — единственный мост между невидимым латентным рассуждением и верифицируемым артефактом.

### 2. CoT не faithful — GRACE не полагается на CoT

GRACE заменяет ненадёжный CoT на формальные артефакты:
- `knowledge-graph.xml` — формальный граф знаний
- `DevelopmentPlan.xml` — компилируемая спецификация
- `MODULE_CONTRACT` — верифицируемые контракты

Это прямое следствие H1: если CoT — лишь проекция, строить на нём интерпретацию решений агента нельзя.

### 3. PCAM переводит ZZ → SS

PCAM (Purpose → Constraints → Autonomy → Metrics) превращает нестабильное латентное рассуждение в верифицируемые поверхностные структуры:

| PCAM | Что делает | Связь с H1 |
|---|---|---|
| Purpose | Формальный контракт (SS) | Фиксирует намерение до того, как ZZ начнёт блуждать |
| Constraints | Границы (SS) | Ограничивает пространство ZZ |
| Autonomy | Агент решает как (ZZ) | Единственный этап, где ZZ работает свободно |
| Metrics | Верификация (SS) | Проверяет, что ZZ пришёл в нужную точку |

### 4. Multi-agent: только XML

Каждый агент имеет свой латентный процесс рассуждения. Текстовая коммуникация между агентами ненадёжна: каждый проецирует своё ZZ в SS по-своему.

GRACE решает это через структурированные XML-отчёты:
```xml
#BLOCK_NAME: CALCULATE_TOTAL_PRICE
#INPUT_DATA: { "quantity": 10, "price": 5 }
#EXPECTED: 50
#ACTUAL: 0
```

XML — общий формат, не зависящий от того, как каждый агент проецирует своё латентное рассуждение.

### 5. Почему grep > vector (дополнительное обоснование)

Статья PwC (arXiv:2605.15184) показала, что grep всегда побеждает векторный поиск на агентах. Статья Wang объясняет **почему**: если рассуждение латентно (ZZ), то агент не может «понять смысл» через embedding'и — ему нужны точные строковые совпадения (SS), за которые и цепляется grep.

GRACE-разметка (уникальные XML-теги, START_BLOCK/END_BLOCK) создана под grep: каждый якорь — уникальная строка для точного поиска. Это не случайность, а следствие того, как реально работают LLM.

## Параллель с психологией

Nisbett & Wilson (1977) показали, что люди тоже не имеют прямого доступа к своим когнитивным процессам и выдумывают объяснения постфактум. CoT в LLM — тот же феномен: модель «объясняет», как она пришла к решению, но это объяснение может не соответствовать реальной траектории в латентном пространстве.

GRACE — единственная методология, которая учитывает это ограничение и не полагается на интроспекцию (ни человеческую, ни машинную).

- Sen et al. (2026). Is Grep All You Need? PwC. arXiv:2605.15184.

# Reasoning Structure of LLMs (ETH Zurich, ICML 2026)

**Источник:** Berdoz et al., "Reasoning Structure of Large Language Models"
**arXiv:** https://arxiv.org/abs/2606.03883
**ICML 2026** — ETH Zurich, DISCO Lab
**Код:** https://github.com/ETH-DISCO/llm-reasoning-efficiency

## Ключевая идея

Авторы не спорят с H1 (латентное рассуждение), а делают **практический инструмент**: берут CoT-текст (SS) и извлекают из него **граф утверждений и зависимостей** — то есть аппроксимируют ZZ через структуру SS.

CoT → граф, где:
- **Узлы** — атомарные утверждения (claims)
- **Рёбра** — дедуктивные зависимости между ними
- **Цвет узла** — валидность: зелёный (верно), красный (ошибка), серый (неверифицируемо), оранжевый (гипотеза)
- **Прозрачность** — статус: непрозрачный (явно stated), полупрозрачный (выведен derived)

## Метрика η (reasoning flow efficiency)

```
η = (log|V| - Hstr(G)) / (log|V| - log|C*|)
```

Где:
- |V| — число узлов в графе
- Hstr(G) — структурная энтропия (разброс логической массы по графу)
- |C*| — минимальное множество утверждений, необходимое для решения

**Интуиция:** η измеряет, насколько концентрирован логический поток относительно минимально необходимого набора утверждений.

- **Высокий η** (→1) — сфокусированное рассуждение, минимум лишних шагов
- **Низкий η** (→0) — диффузное блуждание, много проверок и ответвлений

## Ключевые результаты

1. **Длина CoT ≠ качество.** Два трека одинаковой длины могут иметь η=0.059 (диффузный) vs η=0.362 (сфокусированный). Accuracy и token count не различают их.

2. **Higher efficiency = higher claim accuracy.** Чем концентрированнее логический поток, тем правильнее утверждения.

3. **Higher efficiency = later first-error depth.** Эффективные модели ошибаются позже (или не ошибаются вообще).

4. **Verification overhead растёт с token count.** Больше токенов = больше проверочных утверждений, которые не ведут к решению.

5. **η не коррелирует с verbosity.** Можно быть многословным, но эффективным (редко) или кратким, но бесполезным.

6. **Higher efficiency = higher restated fraction.** Эффективные модели чаще переформулируют已有的 утверждения, чем вводят новые — признак структурированного мышления.

## Бенчмарк PUZZLE

- 21 логическая головоломка (2D grid puzzles: Tents, Islands, etc.)
- 4 уровня сложности: Trivial, Easy, Normal, Hard
- Контролируемое масштабирование difficulty без изменения правил
- Исключает data contamination (свежие задачи)

## Связь с GRACE

### 1. GRACE-логирование как граф утверждений

GRACE structured logging:
```
[DEBUG][load_and_parse_data][PARSE_DATETIME_COLUMNS][Attempt] Парсинг столбцов дат
```

Это **то же самое**, что claim graph из статьи, но на уровне кода, а не рассуждения. Каждый семантический якорь — узел, каждая зависимость между блоками — ребро.

### 2. Belief State = η для кода

Метрика η для рассуждения LLM — аналог того, что GRACE делает для кода: измеряет, насколько сфокусирован агент, а не сколько токенов он потратил.

### 3. Длина ≠ качество — подтверждение GRACE-подхода

GRACE не оценивает качество по объёму кода или числу коммитов. Статья даёт научное обоснование: **структура важнее объёма**. Это же относится к CoT-трассам агента.

### 4. Restated fraction → semantic anchors

Эффективные модели переформулируют已有的 утверждения. В GRACE это соответствует тому, что агент возвращается к существующим семантическим якорям, а не плодит новые — признак зрелого рассуждения.

### 5. Практический инструмент для GRACE

Можно адаптировать pipeline ETH Zurich для GRACE-логов:
- Извлекать claim graph из structured log
- Вычислять η для каждой сессии агента
- Использовать η как метрику качества рассуждения (не accuracy, не token count)

## Вывод

Статья ETH Zurich — **мост между H1 и H2**: CoT не faithful как объяснение, но его **структура** (граф утверждений) содержит информацию о латентном процессе. GRACE делает то же самое для кода: семантические якоря не «объясняют» код, но их **структура** позволяет навигировать по нему.

- Код: https://github.com/ETH-DISCO/llm-reasoning-efficiency

# GLM-5: от Vibe Coding к Agentic Engineering (Zhipu AI + Tsinghua, 2026)

**Источник:** GLM-5 Team, "GLM-5: from Vibe Coding to Agentic Engineering"
**arXiv:** https://arxiv.org/abs/2602.15763
**Дата:** 24 Feb 2026
**Код:** https://github.com/zai-org/GLM-5

## Ключевая идея

GLM-5 — первая open-weight модель, спроектированная под **автономные агентские циклы**, а не под интерактивный чат. Модель обучается на длинных траекториях tool-вызовов (часы автономной работы) через новый алгоритм Asynchronous Agent RL.

**Парадигма:** Vibe Coding (интерактивный режим) → Agentic Engineering (автономные циклы).

## Ключевые инновации

### 1. Asynchronous Agent RL
Новый алгоритм RL, который decouples генерацию от тренировки, позволяя модели учиться на complex, long-horizon interactions. Модель тренируется на циклах: код → тест → код → тест → done, а не на отдельных ответах.

### 2. DSA (Dual-Stage Attention)
Архитектурная оптимизация для long-context fidelity. Позволяет модели работать часами без потери контекста.

### 3. Preserved Thinking между tool calls
Модель сохраняет «мышление» (reasoning state) между вызовами инструментов, не сбрасывая контекст после каждого tool result. Это ключевое отличие от обычных агентов, где каждый tool call — отдельный turn.

### 4. ARC (Agentic, Reasoning, Coding)
Объединение трёх способностей в одной MoE-архитектуре. Не отдельные модели для чата, кода и агента, а одна модель, которая умеет всё.

## Результаты

- **#1 open model** в LMArena Text и LMArena Code
- **SWE-bench Verified: 67.8%** (наравне с Claude Opus 4.5 и GPT-5.2)
- **Intelligence Index v4.0: 50** — первый open-weight model с таким score
- **~20% improvement** над GLM-4.7

## Связь с GRACE

### 1. Подтверждение направления

GRACE изначально спроектирован под автономные циклы: `$grace-execute` → `$grace-verification` → `$grace-fix` → loop. Статья подтверждает, что индустрия движется туда же.

**Цитата из статьи:** «Coding agents can now write code autonomously for hours, and the length and breadth of tasks AI models are able to complete are...»

### 2. Asynchronous Agent RL = тренировочный аналог GRACE

| | GLM-5 (training) | GRACE (inference) |
|---|---|---|
| Цикл | RL на tool-траекториях | $grace-execute → verify → fix |
| Верификация | Reward model | $grace-verification |
| Контекст | Preserved thinking | Structured logging + semantic anchors |
| Масштаб | Часы автономной работы | Многошаговые планы |

То, что GLM-5 делает на этапе обучения, GRACE делает на этапе инференса — один и тот же паттерн на разных уровнях стека.

### 3. Preserved Thinking = Belief State

GLM-5 сохраняет reasoning state между tool calls. В GRACE это structured logging с семантическими якорями — единственный способ сделать это мышление наблюдаемым и верифицируемым.

### 4. Vibe Coding → Agentic Engineering

Та же эволюция, что GRACE описывает: от ad-hoc промптинга к структурированной, контрактно-ориентированной разработке. GRACE — это Agentic Engineering в действии.

### 5. Интерактивный режим — только для спецификаций

Статья подтверждает тезис: интерактивная работа с агентом — для разработки спецификаций и контроля качества. Производственный цикл (код, тесты, документация) — 100% автоматический.

## Вывод

GLM-5 — техническое подтверждение того, что GRACE выбрал правильное направление. Автономные циклы с верификацией — это не нишевый подход, а то, как индустрия будет работать.

## Ссылки

# Практический кейс: ETL-агент для стройки (автор GRACE, 2026)

**Источник:** личный опыт автора GRACE (Владимир Иванов)

## Контекст

Автор заморозил разработку Turbo Planner (legacy ПО для стройки). Все новые решения делаются через ИИ-агентов. Инвестиции в старый софт признаны бесперспективными.

## Решение

Open Source решение для стройки (без лицензии). Ключевая функциональность:

- **AI сам маппит элементы сметы на работы в графике** — без жёстких правил, без ручного маппинга
- Агент автономно выбирает соответствия между двумя гетерогенными схемами данных
- Результат: интеграция смет и графиков за счёт ИИ, а не ручного программирования

## Учебный пример

Из этого production-решения автор сделал учебно-боевой пример для обучения архитектуре агентов для:

- **ETL** (трансформация данных) — агент сам определяет маппинг между source и target схемами
- **DWH** (подготовка данных для отчётов) — агент автономно строит pipeline

## Сдвиг рынка

- Изначально: заказчики из стройки
- Сейчас: банки и технологические компании
- Но внедрения: всё ещё строительные

Агентская технология не привязана к домену — ETL-паттерн одинаков для стройки и финтеха.

## Связь с GRACE

### 1. Legacy → Agentic Engineering

Turbo Planner заморожен. Всё новое — через ИИ. Это прямое подтверждение тезиса GLM-5: Vibe Coding → Agentic Engineering. GRACE — методология для этого перехода.

### 2. Автономный ETL = GRACE-цикл

| Этап | В кейсе | В GRACE |
|---|---|---|
| Input | Смета (XML/таблица) | RequirementsAnalysis.xml |
| Обработка | AI маппит элементы на работы | $grace-execute |
| Верификация | Соответствие сметы графику | $grace-verification |
| Output | Интегрированный график | DevelopmentPlan.xml |

### 3. От конкретного к абстрактному

Автор обрезал production-решение до учебного примера по ETL/DWH. Это GRACE-принцип: сначала решаешь реальную задачу, потом выделяешь паттерн и формализуешь его.

### 4. Доменная независимость

ETL-паттерн одинаков для стройки и банков. GRACE не привязан к домену — методология работает везде, где есть контракты, трансформации и верификация.

## Вывод

Production-кейс, который подтверждает все три теоретические работы в этом документе:
- **Wang (H1)**: агент не объясняет маппинг (CoT), а делает его в латентном пространстве
- **ETH Zurich (η)**: качество маппинга не в длине объяснения, а в структуре соответствий
- **GLM-5**: автономный цикл маппинга — это Agentic Engineering в действии

# Практический кейс 2: ERP Resource Leveling через ИИ-агента (автор GRACE, 2025)

**Источник:** автор GRACE (Владимир Иванов) — ERP/EPM кейс по оптимизации ресурсов

## Проблема: ERP не умеет планировать

ERP расшифровывается как Enterprise Resource Planning, но в реальности «Planning» не работает ни в 1С, ни в SAP. Реально — **Enterprise Resource Facting** (сбор факта). Планирование ресурсов сводится к заявкам, которые считаются в Excel, а не в ERP.

Для проектных компаний (IT, стройка, проектирование) задача называется **Resource Leveling**. Встроенные модули в 1С/SAP неработоспособны. Даже математически корректные системы (Spider Project) требуют ~70 параметров на каждую задачу — и этого мало.

**Ключевые проблемы традиционного подхода:**
- 70 параметров на задачу — неподъёмно для пользователей
- Модели не учитывают реальные ограничения (40 плиточников в санузел, кран не проедет)
- Только ~5% power user могут настроить такую модель
- Проверить модель на корректность крайне сложно
- Результат: Excel «на коленке» + мат при перенабивании в ERP

## Решение: ИИ-агент + MS Project

ИИ-агент интегрируется с MS Project (где Resource Leveling проще, есть параметр «Приоритет»).

**Формула:**
```
Приоритет = F(70 бывших параметров + 50 новых условий)
```

ИИ — семантический процессор. Anthropic доказал, что GPT нативно владеет понятием «интенсивность» в векторах с ~5% погрешностью. Ему доступно манипулировать с цифрами не только математически, но и семантическими операциями с этой точностью.

**Что делает ИИ:**
1. Пробивает по всем задачам **Приоритет 2.0**, за которым спрятано моделирование кучи условий
2. Учитывает реальные ограничения: крану не проехать, по прогнозу дождь → крышу ставим вперёд
3. Объясняет для каждой задачи свою «экспертную оценку»
4. Досчитывает простой resource leveling через Приоритет

**Результат:**
- ИИ убивает 1С/SAP в задаче, где они заявляют себя ERP, но планировать не умеют
- Объём вводимой информации сокращается в десятки раз
- Качество и прозрачность информации резко повышаются
- Работает то, что в старых системах было чистым маркетингом

## Асимметричный сценарий

Это не про «сокращение числа кликов». Это про то, что **с ИИ в чатике работает архиважное директору, а в старых системах — никогда**. ИИ-агент разблокирует кейсы применения ERP, которые ранее были недоступны.

## Связь с GRACE

### 1. PCAM в действии

| PCAM | В кейсе |
|---|---|
| Purpose | Оптимизировать выделение ресурсов с учётом реальных ограничений |
| Constraints | 70+50 параметров, физические ограничения, погода |
| Autonomy | ИИ сам определяет Приоритет 2.0 для каждой задачи |
| Metrics | Resource Leveling через MS Project |

### 2. Примат цели над инструкцией

ИИ не следует жёсткому алгоритму расчёта (который всё равно не работает). Он получает **цель** — оптимизировать ресурсы — и сам выбирает, как это сделать, используя семантическое понимание «интенсивности» и реальных ограничений.

### 3. Belief State = экспертная оценка

ИИ объясняет для каждой задачи свою «экспертную оценку» — это и есть Belief State в действии. Не просто число, а объяснение, почему приоритет именно такой.

### 4. GRACE-цикл

```
RequirementsAnalysis.xml (цель, ограничения)
→ $grace-execute (ИИ вычисляет Приоритет 2.0)
→ $grace-verification (Resource Leveling через MS Project)
→ $grace-fix (корректировка приоритетов)
```

## Смена парадигмы UI: от форм к диалогу

Наличие ИИ в ERP — Game Changer, который меняет структуру вводимой информации. Старые фронты не учитывали наличие «ИИ-оператора».

**Ключевая ошибка:** гипотеза, что для ИИ оптимально делать «как раньше». Но «как раньше» должно сначала работать. Если процесс не работает (как планирование ресурсов сейчас) — продлевать жизнь и фронту, и беку нет смысла.

**Новая реальность:**
- Появляются экспертные параметры (как «Приоритет»), которые человеку сложно поставить
- Такие параметры заполняет **только ИИ-агент**, человек только проверяет
- От фронта не требуется формы ввода — требуются дашборды и визуализаторы, чтобы показать последствия
- «Приоритет» определён семантическими операциями → для ввода нужно **общаться с ИИ**, а не заполнять форму

**Смена use case:**
```
Было:  «заполни форму»
Стало: «обсуди с ИИ → проверь, как ИИ заполнил данные»
```

Концепт старых фронтов с гиперакцентом на формы рассыпается от ИИ, где он сам заполняет данные по результатам обсуждения.

**Связь с GRACE:** это прямое применение PCAM-принципа «примат цели над инструкцией». Форма — инструкция. Диалог — цель. ИИ сам решает, какие данные нужны для достижения цели, и заполняет их. Человек только верифицирует.

## Вывод

ERP-кейс — ещё одно подтверждение, что GRACE-подход (цель вместо инструкций, семантические операции вместо жёстких алгоритмов, Belief State вместо чёрного ящика) работает не только для кода, но и для любых задач, где традиционные системы терпят крах.

### Сравнение: решение автора GRACE vs OpenConstructionERP

**OpenConstructionERP** (https://github.com/datadrivenconstruction/OpenConstructionERP) — open-source ERP для стройки (AGPL-3.0, 380★). Решает ту же задачу — интеграция смет и графиков — но принципиально другим подходом.

| | Решение автора GRACE | OpenConstructionERP |
|---|---|---|
| **Подход** | Агентский ETL: AI сам маппит элементы сметы на работы | Монолитная ERP, 100+ модулей |
| **AI** | Автономный маппинг без жёстких правил | AI-подсказки, решения принимает человек |
| **Лицензия** | Без лицензии | AGPL-3.0 |
| **Установка** | Через Hermes/агентов | `pip install openconstructionerp` / десктоп |
| **Масштаб** | Узкий ETL-паттерн (смета → график) | 48 регионов, 27 языков, 55K+ cost items |
| **Гибкость** | Любая схема данных, агент сам определяет маппинг | Настраиваемые модули, встроенные каталоги |
| **CAD/BIM** | Нет | AutoCAD, Revit, IFC, PDF takeoff |
| **4D/5D** | Только маппинг сметы на график | Полноценное календарное планирование и cost model |
| **Рынок** | Банки + tech (внедрения в стройке) | 100+ стран, строительные компании |

**Вывод:** два подхода к одной проблеме — традиционный ERP с AI-дополнением (OpenConstructionERP) и агентский ETL без жёстких правил (автор GRACE). Первый — готовое решение для стройки «из коробки». Второй — универсальный паттерн, который автор переносит между доменами (стройка → банки → tech). GRACE-подход выигрывает в гибкости, OpenConstructionERP — в покрытии строительной предметной области.

# Agentic RAG (Google Research + Google Cloud, 2026)

**Источник:** Rashtchian, C. & Juan, D.-C. "Unlocking dependable responses with Gemini Enterprise Agent Platform's Agentic RAG"
**URL:** https://research.google/blog/unlocking-dependable-responses-with-gemini-enterprise-agent-platforms-agentic-rag/
**Дата:** 5 Jun 2026
**Статус:** Public preview в Gemini Enterprise Agent Platform

## Ключевая идея

Обычный RAG — одно касание: запрос → поиск → ответ. Если данные размазаны по разным источникам — не повезло.

Google Agentic RAG — **итеративный цикл с верификацией полноты**. Система не останавливается, пока контекст не признан достаточным.

## Архитектура

```
Root Agent → Planner → Query Rewriter → RAG Agent → Sufficient Context Agent → (loop) → Synthesis
```

### Роли агентов

| Агент | Функция |
|---|---|
| **Root Agent** | Парсит запрос, делегирует sub-агентам |
| **Planner** | Разбивает на подобласти, определяет порядок поиска |
| **Query Rewriter** | Превращает сложный запрос в простые подзапросы |
| **RAG Agent** | Ищет по источникам, собирает snippets |
| **Sufficient Context Agent** | Проверяет полноту, при пробелах выдаёт конкретные указания |
| **Synthesis** | Формирует финальный ответ |

### Sufficient Context Agent — ключевая инновация

Не просто «недостаточно данных», а **конкретные указания, чего не хватает и где искать**:

```
Finding: "Есть лекарства и диета."
Gap: "Нет данных об аллергических реакциях."
Feedback: "Ищи rashes и adverse events."
```

Три проверки перед ответом:
1. Есть ли в snippets факты, покрывающие запрос?
2. Достаточно ли черновика ответа?
3. Если нет — что именно отсутствует?

## Результаты

- **FramesQA** (824 вопроса, 2676 PDF): 90.1% cross-corpus accuracy
- **+34%** над vanilla RAG
- Качество не падает с 3 отвлекающими датасетами — Planner отсеивает мусор
- Latency single-corpus ≈ cross-corpus (в пределах 3%)

## Связь с GRACE

### 1. Sufficient Context Agent = $grace-verification

GRACE-цикл `$grace-execute → $grace-verification → $grace-fix` — тот же паттерн, что Google применил к RAG. Разница только в домене:

| | Google Agentic RAG | GRACE |
|---|---|---|
| Цикл | Search → Check → Refine | Execute → Verify → Fix |
| Верификатор | Sufficient Context Agent | $grace-verification |
| Обратная связь | "Ищи X, не хватает Y" | Structured log с gap analysis |
| Условие выхода | Контекст достаточен | Все тесты пройдены |

### 2. Planner + Query Rewriter = PCAM

Planner разбивает запрос на подобласти — это **Purpose** (что ищем). Query Rewriter превращает в подзапросы — это **Constraints** (как именно ищем). RAG Agent исполняет — **Autonomy**. Sufficient Context Agent верифицирует — **Metrics**.

### 3. Итеративный цикл — стандарт, не исключение

Google подтверждает: **один проход — это legacy**. Production-системы должны проектироваться с итеративными циклами и верификацией полноты. GRACE делает то же самое для кода.

### 4. Cross-corpus = multi-source в GRACE

Google решает проблему данных, размазанных по разным хранилищам. GRACE решает ту же проблему для кода: модули, контракты, knowledge graph — те же «островки данных», которые нужно связывать.

## Вывод

Google Agentic RAG — независимое подтверждение GRACE-подхода от одного из крупнейших research-отделов в мире. Sufficient Context Agent — это $grace-verification, перенесённый из кода в RAG. Паттерн один: **итеративный цикл с верификацией полноты**.

# PCAM (Purpose Centric Agent Methodology) и RaR (Rubrics as Rewards)

**Источник:** автор GRACE (Владимир Иванов) + Scale AI
**Статья Scale AI:** Gunjal et al., "Rubrics as Rewards: Reinforcement Learning Beyond Verifiable Domains"
**arXiv:** https://arxiv.org/abs/2507.17746
**Дата:** Oct 2025

## PCAM — Purpose Centric Agent Methodology

**Источник:** Владимир Иванов (автор GRACE) — статья на VK, 11 сен 2025
**URL:** https://vk.com/@turboplanner-metodologiya-pcam-prevraschaem-ai-agentov-iz-rabov-v-partner

Фреймворк техник для адаптации контекста под RL-вознаграждение агентов. Внутри GRACE применяется через **PBS (Purpose Breakdown Structure)** — декомпозицию целей от корневой.

### Ключевой тезис

**LLM имеет приоритетное обучение на цель, а не на инструкции.** Современные LLM создавались не как интерпретаторы инструкций, а как системы, оптимизированные для достижения целей (goal achievers). Их RL-обучение направлено на максимизацию вознаграждения за успешное выполнение задачи.

**Instruction following вреден для многих кейсов**, потому что:
- Если инструкции начинают конкурировать с целями из обучения — LLM склонна к галлюцинациям
- Принуждение к следованию жёстким планам — сознательная деградация возможностей
- Агент лишается возможности найти более оптимальный путь

### 6 принципов PCAM

1. **Примат цели над инструкцией** — каждая задача формулируется через конечную цель. Агент должен понимать «зачем», а не только «что».
2. **Управление через контекстные руководства (Гайды)** — вместо жёстких планов агент получает гайды — структурированные источники знаний и рекомендованных стратегий.
3. **Стандартизация протоколов и инструментов** — жёсткость применяется не к плану действий, а к протоколам коммуникации и интерфейсам вызова инструментов.
4. **Архитектурная масштабируемость через плагинный/сервисный подход** — AI-агент-оркестратор динамически компонует изолированные модули.
5. **Способность к самоизлечению и адаптации** — агент наделяется полномочиями для самостоятельной диагностики и обхода ошибок.
6. **Интегрированный цикл обратной связи** — агенты эксплуатации и разработки в постоянной коммуникации, непрерывное улучшение инструментов и гайдов.

### Ключевая проблема

SFT (Supervised Fine-Tuning) даёт «вырванные из книги страницы»: LLM может написать функцию на Python или спроектировать модули, но строго как один шаг, до ~200 строк. В реальности эти страницы связываются в повествование за счёт **alignment на цель**.

### Решение: рубрики

Контекст дробится на блоки — **рубрики**. Каждая рубрика:
- ≤2000 токенов (лучше 100-200 строк)
- Имеет локальную RL-оценку
- Адаптирована под специфику своего домена

LLM обучается:
1. Сначала декомпозиции контекста на рубрики
2. Потом локальному применению SFT внутри каждой рубрики
3. Потом связыванию рубрик через alignment на цель

### PBS (Purpose Breakdown Structure)

Декомпозиция целей от корневой:
```
Root Purpose
├── Sub-purpose 1
│   ├── Rubric 1.1 (≤2000 токенов)
│   └── Rubric 1.2 (≤2000 токенов)
├── Sub-purpose 2
│   ├── Rubric 2.1
│   └── Rubric 2.2
└── ...
```

Каждый лист — достаточно мал для SFT. Связи между листьями — alignment на корневую цель.

## RaR — Rubrics as Rewards (Scale AI)

Научное подтверждение PCAM-подхода.

### Проблема

RLVR (Reinforcement Learning with Verifiable Rewards) работает для math и code — там есть бинарная верификация. Для реальных задач (медицина, наука) нет одного правильного ответа.

### Решение

Использовать **рубрики как reward signal** вместо бинарной верификации или preference ranking.

Каждая рубрика — набор структурированных критериев (checklist-style supervision), которые:
- Декомпозируют «что делает ответ хорошим» на интерпретируемые подцели
- Дают automatable feedback
- Согласованы с expert intent

### Результаты

- **+31%** на HealthBench над LLM-as-judge baselines
- **+7%** на GPQA-Diamond
- Меньшая variance при использовании маленьких judge-моделей
- Лучшая адаптация к разнообразным форматам evaluation

### Архитектура

```
Rubric → decomposed subgoals → per-subgoal scoring → aggregated reward → RL update
```

Каждый subgoal — проверяемый критерий (не бинарный, но структурированный). Агрегация через взвешенную сумму или LLM-as-judge.

## Связь с GRACE

### 1. PCAM = GRACE-цикл на уровне контекста

| PCAM | GRACE |
|---|---|
| Purpose (корневая цель) | RequirementsAnalysis.xml |
| PBS (декомпозиция) | DevelopmentPlan.xml |
| Рубрики (≤2000 токенов) | Модули, MODULE_CONTRACT |
| Локальная RL-оценка | $grace-verification |
| Alignment на цель | PCAM (Purpose → Constraints → Autonomy → Metrics) |

### 2. Рубрики = семантические блоки GRACE

GRACE-разметка (START_BLOCK/END_BLOCK, MODULE_CONTRACT) — те же рубрики, но для кода. Каждый блок — достаточно мал для SFT-генерации, но связан с общей целью через knowledge graph.

### 3. RaR = научное обоснование $grace-verification

Scale AI доказала, что рубрики как reward signal работают лучше бинарной верификации для не-кодовых доменов. GRACE делает то же самое для кода: `$grace-verification` — это рубрика с набором проверяемых критериев.

### 4. SFT даёт «страницы», alignment даёт «книгу»

LLM может сгенерировать отдельный модуль (SFT), но только alignment на цель через PBS связывает модули в работающую систему. GRACE — методология этого alignment.

## Вывод

PCAM — теоретическая основа GRACE. RaR (Scale AI) — независимое научное подтверждение. Оба говорят об одном: **декомпозиция контекста на верифицируемые блоки + alignment на цель** — единственный способ заставить LLM работать над реальными задачами, а не генерировать «вырванные страницы».

### Дополнительно: эмерджентная топология задач и критериальная оценка гипотез

Автор GRACE отмечает важный эмерджентный феномен: даже без специального обучения GPT склонна вырабатывать **«топологию задач»** — действовать как классификатор, структурируя пространство решений. Это показано в работе «Transformers represent belief state geometry in their residual stream».

**Критериальный метод оценки гипотез** — техника автора GRACE из мозговых штурмов с GPT по спецификациям и научным исследованиям:

1. ИИ генерирует гипотезу
2. Гипотеза оценивается по набору критериев (checklist)
3. Оценка возвращается в цикл для улучшения

Этот метод глубоко нативен для LLM, потому что:
- Он использует ту же архитектуру, что и рубричное RL-обучение
- Оценка качества генерации рубрики делается той же техникой
- Вендоры LLM не рассчитывают только на эмерджентные свойства, а напрямую стимулируют их через критериальные чек-листы

**Связь с GRACE:** `$grace-verification` — это и есть критериальная оценка гипотез агента. Каждый MODULE_CONTRACT — набор критериев, по которым оценивается выполнение. GRACE формализует то, что в LLM возникает эмерджентно.

# DeepSeek V4 mHC: Manifold-Constrained Hyper-Connections

**Источник:** наблюдения автора GRACE при работе с DeepSeek V4 Flash
**Технология:** mHC (Manifold-Constrained Hyper-Connections)

## Проблема: «зомби-режим» LLM

Вендоры LLM для автономности агентов создают длинные SFT-цепочки, где маской давят ответ Tools и заставляют разучить траекторию. Это даёт устойчивость на N шагах, но порождает **«зомби-режим»**: модель игнорирует оператора в середине траектории, пока не доделает своё дело. Оператор для неё — семантический шум.

**Причина:** сигнал, сильно отличный от обучения, затухает от слоя к слою. Нейроны, разученные на зомби-траекторию, дают приоритет токенам из «тихо сам собою я веду беседу». Крик «остановись!» просто не доходит до последних слоёв.

## Решение: mHC (Manifold-Constrained Hyper-Connections)

Гиперконнекции изолируют разные сигналы в разные каналы. DeepSeek V4 выделяет канал под «неожиданные диагностические данные» или «крики оператора» и подключает их напрямую к последним слоям нейросети.

**Эффект:**
- Модель мгновенно реагирует на новый контекст от оператора
- Неожиданные семантические сигналы не затухают, а доставляются до последних слоёв
- Нет «зомби-режима» — можно докричаться в середине tool-траектории
- Ключевое преимущество для debug и тестов: модель должна реагировать на то, чему её не обучали

## Связь с GRACE

### 1. Belief State + mHC = наблюдаемость без потери сигнала

GRACE structured logging с семантическими якорями — способ сделать латентное рассуждение наблюдаемым. mHC — архитектурное решение, которое гарантирует, что сигнал от оператора (или диагностические данные) дойдёт до последних слоёв. Без mHC даже лучший Belief State бесполезен, если модель вас не слышит.

### 2. GRACE-цикл требует отзывчивости

`$grace-execute → $grace-verification → $grace-fix` предполагает, что агент может быть прерван и перенаправлен. «Зомби-режим» делает `$grace-fix` невозможным — модель не слышит указаний, пока не закончит траекторию. mHC устраняет это ограничение.

### 3. DeepSeek V4 Flash на нашей конфигурации

Мы используем `deepseek-v4-flash:cloud` через custom provider. Наблюдения автора GRACE подтверждаются на практике: модель действительно отзывчива на прерывания и новый контекст.

## Вывод

mHC — архитектурное решение проблемы, которую GRACE решает на методологическом уровне. GRACE делает рассуждение наблюдаемым (Belief State), mHC делает модель слышащей (signal isolation). Вместе они дают агента, который и виден, и управляем.

# State First: почему State-архитектуры нативнее для LLM, чем событийные системы

**Источник:** автор GRACE (Владимир Иванов) — гипотеза о нативных механизмах GPT
**Основание:** «Transformers represent belief state geometry in their residual stream»

## База

ИИ движется в «пространстве состояний»: каждый токен-вектор — оценка очередного состояния по набору критериев из обучения. Attention рассматривает цепочку State-векторов целиком и способен уловить сложнейшие корреляции за счёт глобального контекста эволюции State.

## Три нативных механизма GPT

### 1. Формирование структурной модели кода как графа из якорных векторов

Attention легко поддерживает графовые структуры: корреляции между векторами — рёбра графа, векторы высокой размерности аккумулируют семантику. ИИ легко:
- Взять основные сущности алгоритма (классы, функции, таблицы)
- Построить их графовые отношения (вызовы, зависимости)
- Даже на sparse attention

**Связь с GRACE:** knowledge-graph.xml — это и есть граф из якорных векторов. GRACE формализует то, что для LLM нативно.

### 2. Лента машины Тьюринга = эволюция State-графа

ИИ создаёт векторы-токены по одному — это лента машины Тьюринга. Но моделировать так наиболее просто не событийные механизмы, а **эволюцию сложного State**.

**Почему State нативнее событий:**
- **Attention динамический** — оглянувшись на векторы с графом архитектуры, ИИ легко моделирует, как новый State изменил вызовы и влияния компонентов
- **Вероятностная логика** — «механический детерминизм» не обязателен. ИИ легко предполагает: «20% вероятности ошибки такого типа». Вектором моделируются ссылки на другие сущности и гипотезы
- **Глобальный контекст эволюции State** — attention видит всю цепочку State-векторов и улавливает сложнейшие корреляции

### 3. Событийные системы поддерживаются слабо

Асинхронные событийные алгоритмы нативно поддерживаются LLM плохо. Attention не оптимизирован для отслеживания цепочек событий, размазанных по времени и контексту.

## Рекомендация: State First

При проектировании архитектуры для работы с ИИ:

1. **Сначала хорошее графовое представление** целевой системы (knowledge-graph.xml)
2. **Потом чёткое понятие глобального State** — сложность State не играет большой роли
3. **Событийные системы — только там, где без них нельзя**

**Дополнительные преимущества State First:**
- Лёгкая транзакционная целостность данных
- Лёгкая верификация человеком — не по коду, а по данным о State

## Связь с GRACE

GRACE — методологическая реализация State First:
- `knowledge-graph.xml` — графовое представление системы (п.1)
- `DevelopmentPlan.xml` — глобальный State архитектуры (п.2)
- `MODULE_CONTRACT` — контракты, определяющие эволюцию State
- Belief State — observable state LLM в каждый момент генерации
- Structured logging — лента эволюции State, привязанная к семантическим якорям

## Вывод

State First — не просто архитектурный паттерн, а следование нативной природе LLM. GRACE — методология, которая это реализует. Событийные системы с ИИ будут всегда проигрывать State-ориентированным в предсказуемости, верифицируемости и качестве генерации.
